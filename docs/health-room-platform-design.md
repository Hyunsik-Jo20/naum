# 보건실 디지털 전환 플랫폼 — 설계 확정본 v1.1

> 작성일: 2026-06-13
> 상태: **확정본 v1.1** (1단계 MVP 구현 착수 기준 · AI 활용 섹션 추가)
> 코드명(가칭): **나음(NaUM)** — "나아서 (교실로) 나간다" / 명칭 검토 예정

---

## 1. 개요 — 한 줄 정의

학생이 보건실에 오면 **본인이 직접 증상을 접수**하고, 보건교사는 **처치에만 집중**하며,
담임·학부모는 **접수·종료 상태를 실시간으로** 확인하는, **개인정보가 서버에 남지 않는** 보건실 운영 플랫폼.

---

## 2. 문제 정의

| 현재 | 문제 | 결과 |
|---|---|---|
| 보건교사가 학생 이름·증상·처치를 모두 수기 입력 | 치료자와 기록자가 동일인 → 동시 수행 불가 | 몰릴 때 누락, 하교 후 기억으로 사후 입력 |
| 기록 시점이 "사후 기억" | 실제 상황과 기록 불일치 빈번 | 통계·법적 근거·민원 대응 신뢰도 저하 |
| 학생 위치 정보 공백 | 교실→보건실→귀가 사이 학생 행방 불명 | 안전 우려, 교사·학부모 불안, 민원 |

**핵심 통찰:** 문제의 본질은 보건교사의 태만이 아니라 **"기록의 시작점이 사후 기억"**이라는 구조다.
→ 기록 시작점을 **"실시간 본인 진술(셀프 접수)"**로 이동시키는 것이 해결의 본질.

## 3. 범위 & 로드맵

### 3.1 1단계(MVP) 범위 — 단일 학교
- **포함:** 키오스크 셀프 접수 / 보건교사 처치기록(병명 12계통) / 담임·학부모 실시간 알림(접수·종료 2회) / 월간 교장 보고서 자동 생성 / (선택) 증상→병명·계통 AI 추천 — API 키 입력 시 활성 (15장)
- **제외(후속):** 교육청 다중 학교 집계, 감염병 신호 탐지, 날씨·지형 연계 예측
  - → 단, **데이터 구조는 후속 기능 자리를 미리 비워두고 설계**(스키마 변경 없이 확장).

### 3.2 단계별 로드맵
| 단계 | 범위 | 핵심 가치 |
|---|---|---|
| **1 (MVP)** | 키오스크 접수 + 처치기록 + 담임/학부모 알림 + 월간 보고서 자동 | 기록 신뢰성 + 위치 공백 해결 |
| 2 | 단일교 통계 고도화, 반복방문 패턴, 처치 템플릿 강화 | 보건교사 업무 경감 |
| 3 | 교육청 다중학교 비식별 집계, 카카오맵, 감염병 신호, 양방향 공지 | 공중보건 |
| 4 | 날씨·지형 외부데이터 연계 예방·예측 | 선제 대응 |

## 4. 설계 원칙

1. **개인정보 로컬 격리 (Privacy by Design) — 최우선.** 학생 PII(이름·학번·반·번호·연락처)는 **보건실 로컬에만**. 서버는 **비식별 데이터만**.
2. **키오스크는 기본 경로, 보건교사 우회는 항상 가능.** 응급·저학년·미입력 상황에서 즉시 대리 접수/입력. 단독 의존 금지.
3. **작게 시작, 자리는 비워둔다.** MVP는 단일 학교. 단 데이터·코드 체계는 다중 학교/교육청 집계가 가능한 형태로.
4. **선택이 입력을 이긴다.** 증상은 자유 텍스트보다 **코드화된 선택지** 우선. 자유 입력은 보조.
5. **상태는 곧 위치다.** 접수/대기/처치중/종료 상태가 곧 "학생이 지금 어디 있는지"를 의미.
6. **무문자(無文字) 단순성 — "글자 못 읽어도, 3번 누르면 끝".** 기본 동선은 **QR 찍기 → 그림 누르기 → 완료**. "쓰기"는 전부 폴백.
7. **기본값이 일을 줄인다.** 가장 흔한 경우는 아무것도 안 눌러도 처리(예: 결과 미선택 → 자동 교실 복귀).

## 5. 시스템 아키텍처 (개인정보 분리)

```
┌───────────────────────── 학교 내부 (로컬) ─────────────────────────┐
│  [키오스크] ──접수──▶ ┌────────────────────────┐                    │
│  학생 셀프입력         │  로컬 스테이션 (보건실)  │                    │
│                        │  - 학생 명부 (PII 보유)  │                    │
│  [보건교사 대시보드]◀─▶│  - visit_id ↔ 학생 매핑  │                    │
│  처치 기록             │  - 담임/학부모 토큰 매핑 │                    │
│                        └───────────┬────────────┘                    │
└────────────────────────────────────┼────────────────────────────────┘
                                     │ 비식별 이벤트만 송신
                                     │ (visit_id, 학년, 성별, 계통, 처치, 상태, 시각)
                                     ▼
                     ┌─────────────────────────────┐
                     │     클라우드 플랫폼 서버       │  ← 개인정보 0
                     │  - 비식별 방문 기록 저장      │
                     │  - 알림 토큰 라우팅(이름 모름)│
                     │  - 통계/보고/교육청 집계      │
                     └───────────────┬─────────────┘
                                     │ push (이름 없는 페이로드)
                        ┌────────────┴────────────┐
                        ▼                          ▼
                 [담임 교사 앱]             [학부모 앱]
                 기기 내 명부로 이름 표시    자기 자녀로 표시
```

### 5.1 데이터가 어디에 사는가
| 데이터 | 로컬(보건실) | 서버 | 담임 앱 | 학부모 앱 |
|---|:---:|:---:|:---:|:---:|
| 이름·학번·반·번호 | ✅ | ❌ | (담당 학급만, 기기 내) | (본인 자녀만) |
| 담임/학부모 연락처·푸시토큰 | ✅ | ❌(토큰만, 익명) | — | — |
| 학년 / 성별 | ✅ | ✅ | ✅ | ✅ / — |
| 병명 계통 / 처치 / 상태 / 시각 | ✅ | ✅ | ✅ | ✅ |
| visit_id (난수 토큰) | ✅(매핑 보유) | ✅(이름 모름) | ✅ | ✅ |

→ **서버 DB가 통째로 유출돼도 "어느 학년·성별이 무슨 계통이었다"까지만. 누구인지는 불가.**

### 5.2 알림 라우팅 — "이름 없이 어떻게 보내나"
**식별은 로컬, 라우팅은 익명 토큰.**
1. 담임·학부모가 앱 설치 시 기기마다 **익명 푸시 토큰** 발급 → 로컬 스테이션에 등록(서버는 토큰이 누구 것인지 모름).
2. 접수 시 로컬이 `학생 → 담임/학부모 토큰` 조회 → 서버에 "**토큰 T에게 visit_id V가 상태 S**"만 지시.
3. 서버는 이름 없는 푸시 전송. 학부모는 자기 자녀임을 이미 앎(이름 불필요), 담임 앱은 **기기 내 학급 명부**로 이름 표시(서버 미경유).

### 5.3 로컬 ↔ 서버 조인 키 (확정)
**키는 실제 학반+번호가 아니라 방문별 난수 토큰(visit_id).**
- 학반+번호는 단일 학교에서 학생 1명을 특정 → 반 명렬표(비밀 아님)와 대조 시 재식별(가명정보). 매년 반배정으로 불안정 → **키로 부적합.**
- 방문마다 `visit_id`(난수) 생성, 로컬만 `visit_id ↔ 학생` 매핑 보유. 서버엔 학년·성별·계통·시각만.
- **토큰 수명 = 방문별(per-visit)만.** 같은 학생도 방문마다 새 난수 → 서버는 동일 학생 방문을 못 엮음(최대 익명). **반복 방문 파악은 로컬에서** 이름으로 수행.
- (보류) 서버 차원 익명 반복 분석이 필요하면 3단계에서 영속 `student_token`(난수, 로컬 매핑) 재판단. MVP 제외.

## 6. 핵심 사용자 흐름

### 6.1 접수 → 종료 전체 흐름 (병원 외래 모델)
```
[접수]        [대기]        [호출/처치중]     [처치완료]      [종료]
  │            │                │               │            │
학생 키오스크  대기열 표시     보건교사 호출    처치 입력     교실 복귀/귀가/병원
증상 셀프선택  (위치 가시화)   처치 시작        결과 처리     상태 전송
  │            │                │               │            │
  └─visit 생성─┴──상태 갱신─────┴───────────────┴────────────┘
       알림은 2회만: ① 접수 시 / ② 종료 시 (담임·학부모 동일)
       중간 단계(처치 중 등)는 알림 없음 — 알림 피로 최소화
```

### 6.2 학생 키오스크 접수 — 신원 + 증상, 다중 입력 수단
접수는 **신원 확인**과 **증상 입력** 두 단계. 각 단계 수단을 다중화하고 실패 시 폴백.

**단계 A — 신원 확인 (우선순위 순)**
1. **QR 스캔 (1순위):** 이름표·학생증의 QR. 타이핑 0, 저학년 가능, ~0.5초. QR엔 **이름이 아니라 로컬 난수 ID만** → 분실해도 누구 건지 모름.
2. **이름 선택 (폴백):** 반 → 이름 터치. **사진·번호 병행**으로 저학년 보조.
3. **보건교사 대리 (최종 폴백).**

**단계 B — 증상 입력 (다중 선택 지원)**
1. **그림(아이콘) 선택 (1순위):** 큰 그림 타일. **글 못 읽어도 누름.** **토글로 여러 개 선택**(예: 배 아프고 어지러움) → 개수 표시 → **"다 골랐어요"로 확정**.
2. **음성(마이크) (보조):** 그림으로 부족할 때 한마디. ⚠️ 음성=민감정보 → **로컬에서만 STT, 텍스트 변환 후 음성파일 즉시 폐기, 서버 전송·저장 금지.**
3. **텍스트 입력 (보조, 고학년·교사용).**
4. **보건교사 대리 (최종 폴백).**

**공통:** 완료 → 대기번호 발급 + 담임·학부모 접수 알림. 응급/거동불가 시 "도움 요청" 큰 버튼 또는 보건교사 직접 접수. 목표 동선 **QR → 그림 → 완료(3탭 이내, 무문자).**

### 6.3 보건교사 처치 — 입력 최소화
- 대기열에서 학생 호출 → 학생이 고른 증상이 그대로 표시(재입력 불필요).
- **병명 다중 확정** → 계통 12종 **자동 분류**, 대표 1개를 **주증상(★)** 지정.
- **처치 다중 선택**(자주 쓰는 템플릿 칩) + 직접 입력은 보조.
- **결과(outcome) 규칙 (확정):**
  - **기본값 = 교실 복귀.** **결과를 선택하지 않고 처치 완료 시 자동으로 "교실 복귀" 처리** → 가장 흔한 경우 무입력.
  - **귀가** 선택 시 → *보호자 연락·인계 확인* 추가.
  - **병원 이송** 선택 시 → *동행자(보건교사·담임·교감·교장·보호자·119) + 이송 방법(자가·119)* 추가 → 이송 책임 소재·안전 기록 확보.
  - (선택) 계속 관찰.
- 모든 시각(접수·호출·처치·종료)은 **자동 캡처** → 수기 시각 입력 폐지(사후 입력 구조적 차단).

## 7. 어휘 & 코드 체계 (실제 일지 8개월 분석 기반)

### 7.1 어휘 3계층 (핵심)
```
[학생 화면]        [보건교사]       [통계·보고]
쉬운 말+그림  →    구체 병명   →    12계통(고정)
"다쳤어요🩹"        찰과상           피부피하계
"배 아파요🤢"       복통             소화기계
  (학생 선택)      (교사 확정)     (자동 롤업)
```
학생은 **어려운 용어(예: 피부피하계) 대신 쉬운 말+그림**으로 선택. 계통 변환은 시스템·교사가 처리.

### 7.2 통계 계통 = 12개 고정 (전 8개월 동일, 확정)
```
1 호흡기계  2 소화기계  3 순환기계  4 정신신경계  5 근골격계  6 피부피하계
7 비뇨생식기계  8 구강치아계  9 이비인후과계  10 안과계  11 감염병  12 기타
```
이 12계통이 통계·월간 보고서의 **집계 축**(고정). KCD 매핑 필드는 후속 확장용으로만 확보.

### 7.3 구체 병명 사전 (실데이터 시드 — 개방형)
보건교사는 구체 병명을 선택(또는 추가)하고 계통은 자동 롤업(확인·수정 가능). 빈도는 8개월 실측.

| 병명(예, 빈도) | → 계통 | 병명(예, 빈도) | → 계통 |
|---|---|---|---|
| 찰과상(152)·외상(15)·벌레물림(29)·두드러기(5)·단순포진(3)·화상(2)·가시박힘·모기물림 | 피부피하계 | 복통(68)·식체(21)·소화불량(3)·속쓰림 | 소화기계 |
| 타박상(59)·염좌(44)·근육통(25) | 근골격계 | 두통(71)·어지러움(6) | 정신신경계 |
| 비출혈(12)·귓병 | 이비인후과계 | 인후통(9)·편도염·비염·기침 | 호흡기계 |
| 충혈(8)·눈다래끼·눈이물삽입 | 안과계 | 구내염(3)·치통·치아흔들림 | 구강치아계 |
| 생리통(2)·생리대(보건용품) | 비뇨생식기계 | 분류 애매·기타(78) | 기타 |

- **최초 도입 시 보건교사가 사전을 1회 검수**(학교별 관행 반영). 신규 병명 추가 시 계통 지정.

### 7.4 학생 타일 = 12계통의 쉬운 말 변환 (키오스크)
| 12계통 | 학생 타일(쉬운 말 + 그림) | 노출 |
|---|---|---|
| 피부피하계 | 다쳤어요·긁혔어요·벌레 물렸어요 🩹 | ✅ |
| 소화기계 | 배 아파요·토할 것 같아요 🤢 | ✅ |
| 근골격계 | 삐었어요·팔다리 아파요 💪 | ✅ |
| 정신신경계 | 머리 아파요·어지러워요 🤕 | ✅ |
| 호흡기계 | 기침·콧물·목 아파요 🤧 | ✅ |
| 이비인후과계 | 귀 아파요·코피 나요 👂 | ✅ |
| 안과계 | 눈이 아파요·가려워요 👁️ | ✅ |
| 구강치아계 | 이가 아파요·입 안이 아파요 🦷 | ✅ |
| 비뇨생식기계 | 화장실이 불편해요(+생리) | △ 민감 → 축약/별도 동선 |
| 순환기계 | 가슴이 두근거려요 | △ 드묾 → "가슴" 부위로 흡수 |
| 감염병 | (학생이 못 고름) | ✕ 보건교사 판단 분류 |
| 기타 | 잘 모르겠어요·그 밖에 ❓ | ✅ (필수) |

**결론:** 학생 화면 = **흔한 8개 그림 타일 + "잘 모르겠어요"**. 드물거나 민감한 계통(감염병·순환기계·비뇨생식기계)은 학생 타일에서 빼고 **보건교사 확정 단계에서 분류** → 화면 단순, 통계 12계통 완전 커버.

### 7.5 처치 템플릿
`응급처치 / 투약 / 관찰 / 안정 / 교육 / 의뢰(보호자연락·병원)` 분류 + 자주 쓰는 항목 버튼
(예: 밴드·후시딘 / 얼음찜질 / 지혈 / 안약점안 / 체온측정 / 꼬마활명수).

### 7.6 다중 선택 & 집계 규칙 (확정)
- 학생 증상·병명·처치 모두 **다중 허용**.
- **"방문 수 ≠ 병명 건수":** 한 방문에 병명 2개면 두 계통에 각각 1씩 집계 → 계통 합계 총합 ≥ 방문 수.
- **월간 보고서:** `방문 학생 수(N명)` + `병명 건수(M건)` 분리 표기.
- **주증상(is_primary):** 대표 병명 1개 지정 → 기존 종이일지식 "방문당 1건" 보고가 필요하면 주증상만 집계하는 모드도 지원(양쪽 호환).

## 9. 산출물 자동화 — 실제 보건일지 → 시스템

실제 일지(부산, 1일 단위) 구조와 매핑:
1. **머리말** (결재란·날짜·요일·날씨) → `DailyLog` + 전자결재
2. **일일 요약** (보건교육·보건행사·병원치료·특이사항) → `DailyLog` 자유서술(로컬)
3. **응급처치 표** (연번·학년반·성명·성별·병명·처치) → `Visit` (성명·반은 로컬, 시각 자동)
4. **통계표** (일계/월계/누계 × 성별 × 12계통) → `Visit` 집계로 **자동 생성**

**핵심 가치:** 통계·월간 교장 보고서가 **서버 비식별 데이터만으로 자동 생성** → 수기 집계 폐지.
**개인정보를 서버에 안 보낸다는 제약이 보고 자동화와 완벽히 맞물림** = 최대 셀링포인트.
결재란은 **전자결재(계→부장→교장)**로 디지털화.

## 11. (3단계) 교육청 대시보드 & 학교 ↔ 교육청 공지

> MVP 제외, **3단계**(다중 학교 도입 후 가치 발생). 서버가 이미 보유한 비식별 데이터로 구현 → 데이터 추가 불필요, 지금은 자리만 확보.

- **데이터:** 비식별 집계(`학교·학년·성별·계통·시각`)만. 드릴다운 시 **소수셀 억제(5명 미만 숨김)/k-익명성**으로 소규모 학교 재식별 방지.
- **필터:** 기간 / 지역(구·군) / 학교급(초·중·고) / 학교 / 병명계통 / 학년·성별
- **주요 화면:**
  - 보건실 이용 현황: 지역·학교별 방문 추이(일/주/월)
  - 병명별(계통별) 현황: 분포 · 증가 상위 계통
  - 이상 신호(조기 경보): 특정 계통 급증/감염병 임계 초과 학교 강조
  - 학교 비교: 학생 수 대비 방문율 등 정규화 지표
  - **지도(카카오맵 연동):** Kakao Maps JS SDK로 관내 학교 위치 마커, 이상 신호 학교 강조, 마커 클릭 시 비식별 요약(필터 연동)
- **공지(양방향):** 교육청 → 학교(예방수칙·계절질환·감염병 주의) / 학교 → 교육청(보고·회신·지원 요청)
- **가치:** 감염병 조기 신호, 지역 단위 보건 지원, 선제 예방 안내. (날씨·지형 연계 예측은 4단계)

## 14. AI 활용 (선택적 · API 키 연동)

### 14.1 활성화 방식 — BYO API Key
- AI 기능은 **기본 비활성.** 설정 화면에서 **API 키를 입력하면 활성화**(Bring-Your-Own-Key).
- 키가 없으면 해당 기능 숨김 + 수동 모드로 **graceful degradation** — 플랫폼 핵심 기능은 AI 없이도 완전 동작.
- 키 저장: 보건교사용은 **로컬 스테이션**, 교육청용은 **교육청 서버**에 암호화 보관.
- 모델: 가볍고 빠른 작업은 `claude-haiku-4-5`, 품질이 중요한 작성·브리핑은 `claude-opus-4-8`. 설정에서 선택.

### 14.2 개인정보 가드레일 (필수)
- 클라우드 LLM 프롬프트에 **이름·반·번호 등 PII 절대 미포함.**
- 보건교사 AI: 전송 전 **비식별화**(증상·병명·계통 텍스트만). 향후 온디바이스/로컬 모델 옵션 검토.
- 교육청 AI: 비식별 집계만 다룸 → 구조적으로 안전.
- 모든 AI 출력은 **사람이 확인·수정·승인(human-in-the-loop).** 진단·자동 발송 위임 금지.

### 14.3 보건교사 대시보드 AI (MVP 포함)
- **증상 → 병명·계통 자동 추천:** 학생 선택 증상(+자유/음성 텍스트) → 후보 병명 + 12계통 매핑 제안. 보건교사 확인/수정. 입력 최소화의 핵심.

### 14.4 교육청 대시보드 AI (3단계)
- **이상 신호 탐지 + 설명:** 계통별 급증·임계 초과 자동 포착 + 원인 가설·권고 행동을 자연어로 요약.
- **자연어 질의:** "사하구 초등 이번 달 호흡기 추이" → 필터/쿼리 변환 + 결과 요약.
- **공지문 초안 자동 작성:** 신호 기반 예방수칙 공지 초안 생성 → 담당자 수정·발송.
- **주간 지역 브리핑 자동:** 비식별 집계 → 자연어 요약 리포트 정기 생성.
